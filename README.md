# Платформа обработки задач

Модульная платформа для приема, хранения и асинхронной обработки задач из различных источников.


## Структура проекта

```
lab_5/
├── src/                                    # Исходный код
│   ├── async_data/                         # Асинхронные модули
│   │   ├── handlers/                       # Обработчики задач
│   │   │   ├── async_handler_protocol.py   # Контракт обработчика
│   │   │   └── async_handlers.py           # Реализации обработчиков
│   │   ├── context_mgmt.py                 # Контекстные менеджеры
│   │   ├── dispatcher.py                   # Диспетчер задач
│   │   └── worker.py                       # Асинхронный воркер
│   ├── protocols.py                        # Контракты задач и источников
│   ├── task_module.py                      # Приемник задач с runtime-проверкой
│   ├── task.py                             # Модель задачи
│   ├── task_collections.py                 # Очередь задач (асинхронная)
│   ├── task_examples.py                    # Примеры задач
│   ├── task_source_examples.py             # Примеры источников
│   ├── constants.py                        # Константы
│   └── main.py                             # Демонстрация работы
├── tests/                                  # Unit-тесты
└── README.md
```

## Лабораторная работа №1. Источники задач и контракты

### Реализованные компоненты

Контракты (`protocols.py`)
- `TaskProtocol` - контракт для задачи (поля `id`, `payload`)
- `TaskSourceProtocol` - контракт для источника задач (метод `get_tasks()`)

Приемник задач (`task_module.py`)
- `TaskModule` - класс для приема задач из различных источников
- Runtime-проверка соответствия протоколам через `@runtime_checkable`
- Поддержка задач в формате `dict`, `NamedTuple`, `dataclass`, пользовательских классов

Примеры источников (`task_source_examples.py`)
- `FileTaskSource` - загрузка задач из JSON/CSV файла
- `GeneratorTaskSource` - программная генерация задач
- `APIMockTaskSource` - имитация получения задач из внешнего API

### Ключевые особенности
- Источники не связаны наследованием, только контрактом
- Расширяемая архитектура - новые источники добавляются без изменения кода
- Проверка через `isinstance()` и `issubclass()`
- Полная типизация публичных интерфейсов


## Лабораторная работа №2. Модель задачи: дескрипторы и @property

### Атрибуты класса Task

- `id` (str / int, чтение/запись) - уникальный идентификатор
- `payload` (Any, чтение/запись) - полезная нагрузка
- `description` (str, чтение/запись) - описание задачи, по умолчанию пустая строка
- `priority` (int, чтение/запись) - приоритет от 0 до 10
- `status` (str, чтение/запись) - статус задачи: created, started или finished
- `planned_time` (timedelta / int / float, чтение/запись) - плановое время выполнения
- `created_at` (datetime, только чтение) - время создания задачи
- `started_at` (datetime, только чтение) - время начала выполнения
- `finished_at` (datetime, только чтение) - время завершения выполнения
- `duration` (timedelta, только чтение) - длительность выполнения
- `is_ready` (bool, вычисляемое) - признак готовности задачи к выполнению
- `is_valid` (str, вычисляемое) - информация о заполненности атрибутов

### Дескрипторы

Data-дескрипторы (с `set`):
- `IdAttr` - валидация уникальности ID
- `PriorityAttr` - проверка диапазона 0-10
- `PlannedTimeAttr` - приведение к timedelta
- `StatusAttr` - контроль переходов, установка таймеров
- `CreatedAtAttr` - только чтение
- `DurationAttr` - вычисление длительности

Non-data дескрипторы (без `set`):
- `IsReadyAttr` - проверка готовности задачи
- `IsValidAttr` - анализ заполненности атрибутов

### Валидация состояния
- Глобальное множество _all_ids для контроля уникальности ID
- Запрет добавления новых атрибутов через __setattr__
- Автоматическая установка таймеров при смене статуса
- Специализированные исключения при нарушении инвариантов

## Лабораторная работа №3. Очередь задач: итераторы и генераторы

### Возможности TaskQueue

- Итерация:
```
for task in queue:           # Прямой обход
for task in reversed(queue): # Обратный обход
```

- Доступ по индексу:
```
task = queue[0]              # Получение элемента
queue[0] = new_task          # Изменение элемента
del queue[0]                 # Удаление элемента
slice = queue[1:3]           # Срез (создает новую очередь)
```

- Ленивый фильтр
```queue.filter_by(status="finished", priority=5)```

- Операции сложения (+, +=):

Task - добавить одну задачу
TaskQueue - объединить две очереди
list[Task] - добавить список задач

- Операции вычитания (-, -=):

Task - удалить конкретную задачу
TaskQueue / list[Task] - удалить набор задач
int - удалить задачу по ID
callable - удалить по условию: ``` queue - (lambda t: t.priority < 3) ```

- Прочие методы:

``` len(queue) ``` - количество задач
``` task in queue ``` - проверка вхождения
``` queue.is_empty() ``` - проверка на пустоту

- Обработка ошибок

TaskQueueException при неверном типе
StopIteration при модификации во время итерации

## Лабораторная работа №4. Асинхронный исполнитель задач

### Основные компоненты

#### AsyncTaskQueue - асинхронная очередь задач

- Конкурентный доступ через asyncio.Condition
- Асинхронные методы put() / get() с ожиданием на пустой очереди
- Graceful shutdown через close()
- Экстренное прерывание очереди через terminate()
- Поддержка async for с помощью __aiter__ / __anext__
- Асинхронный фильтр async_filter_by()
- Синхронный метод add() для совместимости с синхронным кодом

#### AsyncHandler Protocol - контракт обработчика

#### Dispatcher - маршрутизатор задач:

- Принимает список обработчиков при инициализации
- Последовательно проверяет can_handle()
- Вызывает первый подходящий обработчик
- Выбрасывает исключение если обработчик не найден

#### Worker - асинхронный воркер:

- Бесконечный цикл обработки задач из очереди
- Централизованная обработка ошибок
- Логирование сбоев без остановки

#### Обработчики:

- PrintHandler - логирование задачи
- DatabaseHandler - сохранение в БД (использует менеджер DatabaseSession)
- HttpPostHandler - HTTP POST запрос (использует менеджер HTTPSession)

#### Асинхронные контекстные менеджеры:

- DatabaseSession - управление соединением с __aenter__ / __aexit__
- HTTPSession - управление HTTP-сессией, методы get() / post()

### Ключевые особенности

- Параллельная обработка задач несколькими воркерами
- Отсутствие блокирующих операций в event loop
- Расширяемая архитектура - новые обработчики добавляются без изменения существующего кода
- Корректное освобождение ресурсов через async with
- Централизованное логирование всех этапов обработки
- Полная типизация публичных интерфейсов
